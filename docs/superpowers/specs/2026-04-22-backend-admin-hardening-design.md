# 后端与管理后台安全增强及 CRUD 补齐设计

## 文档说明

- 日期：2026-04-22
- 覆盖范围：`demo/` Spring Boot 后端、`demo_ui/` 管理后台
- 文档落点说明：工作区根目录不是 git 仓库，因此本设计文档存放在已纳管的 `demo/` 仓库内，内容同时覆盖后端与管理后台改造

## 背景

当前系统已经具备基础登录、管理员后台、预约订单、意见反馈、动态发布、机构管理等能力，但存在以下问题：

1. 管理后台登录验证码由前端直接调用第三方验证码 API，验证码生成与校验都暴露在前端，安全边界不合理。
2. 登录与注册没有统一的后端验证码风控策略，无法阻断脚本批量尝试。
3. `ADMIN` 仍可走自助注册路径，不符合后台账号管理习惯。
4. 首个管理员虽然可由系统初始化，但当前没有“首次登录强制改密”机制。
5. 预约订单、意见反馈、动态发布在后台侧都只具备部分 CRUD，无法满足完整管理需求。
6. 机构管理页操作按钮较多，当前布局容易换行，降低管理效率。

## 目标

本次改造目标如下：

1. 将管理后台登录、注册验证码能力统一收回后端。
2. 使用当前前端已接入的第三方验证码服务，但由后端代理生成、代理展示、后端校验。
3. 对验证码错误增加 60 秒冷却限制，阻断脚本高频重试。
4. 移除 `ADMIN` 自助注册，仅保留普通用户自助注册。
5. 首个 `ADMIN` 由系统初始化提供，默认密码固定为 `200502`，首次登录必须修改密码。
6. 为预约订单、意见反馈补齐后台完整增删改查，其中删除采用逻辑删除。
7. 为动态发布补齐后台查看与编辑链路，删除保留物理删除。
8. 优化机构管理页操作区，保证按钮固定单排展示，宽度不足时通过表格横向滚动而不是按钮换行。
9. 实施阶段严格执行“每次代码改动后立即跑一次对应测试或编译校验”。

## 非目标

本次不包含以下内容：

1. 不接入新的验证码供应商，继续使用现有第三方验证码 API。
2. 不引入短信验证码、滑块验证码或人脸验证。
3. 不建设完整的账号锁定中心、回收站页面或复杂审计平台。
4. 不改动小程序登录注册链路，除非与共享后端实体或公共逻辑存在必要耦合。

## 总体方案

本次采用“安全优先但范围收敛”的方案：

1. 认证安全增强聚焦于“验证码后移到后端、验证码错误冷却、后台管理员入口收紧、首次登录强制改密”。
2. 后台业务增强聚焦于“预约订单、意见反馈、动态发布”的管理能力补齐。
3. 页面优化聚焦于“登录注册交互、强制改密流程、后台 CRUD 操作体验、机构管理操作区布局”。
4. 后端新增能力优先复用现有基础设施：Redis 负责验证码短时态会话，MySQL 负责用户与业务实体持久化，现有 JWT 鉴权继续沿用。

## 当前系统现状摘要

### 认证现状

1. 管理后台前端当前直接请求第三方验证码接口并在前端校验。
2. 后端 `POST /api/login` 只校验用户名和密码，不接收验证码参数。
3. 后端 `POST /api/register` 允许带 `role` 的自助注册，不适合后台管理员体系。
4. 默认管理员由 `DataInitializer` 初始化，但密码来自配置哈希，且无首次改密强约束。

### 业务现状

1. 预约订单已有创建、列表、状态变更、派单等部分能力，但缺少后台统一详情、编辑、删除。
2. 意见反馈后台只有列表和状态修改，缺少详情接口、编辑接口、新建接口和删除接口。
3. 动态发布后端已有创建、列表、详情、更新、删除能力，但后台页面仅完成部分管理交互。
4. 机构管理页面操作较多，当前在表格里容易拥挤或换行。

## 架构设计

### 1. 认证与验证码架构

管理后台认证链路调整为：

1. 前端请求后端获取验证码会话。
2. 后端调用现有第三方验证码 API 生成验证码。
3. 后端将第三方验证码标识映射为本地 `captchaToken`，并把会话写入 Redis。
4. 前端仅使用后端返回的 `captchaToken` 和后端提供的图片地址展示验证码。
5. 登录或注册提交时，前端把 `captchaToken + captchaCode + 表单字段` 一并提交给后端。
6. 后端统一完成冷却校验、会话校验、第三方验证码校验、业务校验。

这样做后：

1. 第三方接口地址、第三方验证码 ID、第三方校验逻辑都不会暴露给前端。
2. 登录与注册都能共用一套风控策略。
3. 后续即使更换验证码服务商，前端也无需重写交互协议。

### 2. 首个管理员与首次改密架构

首个管理员初始化策略：

1. 用户名固定为 `admin`。
2. 默认密码固定为 `200502`。
3. 系统启动时若不存在 `id=1000` 或用户名 `admin`，则自动创建该管理员。
4. 该管理员创建时写入 `force_password_change = true`。

首次登录流程：

1. 登录成功后，后端在响应中返回 `forcePasswordChange`。
2. 前端收到该标记后，强制跳转到改密页。
3. 改密完成前，只允许访问改密接口、当前用户信息接口和退出登录接口。
4. 改密成功后，后端清除强制改密标记并更新时间戳，前端再放行正常路由。

### 3. 后台业务管理接口分层

本次将后台管理接口与通用业务接口进一步分层：

1. 面向后台管理的接口统一放在 `/api/admin/**` 下。
2. 现有面向小程序或通用展示的接口保留原职责，不强行复用后台查询逻辑。
3. 后台接口可返回更完整的管理字段，例如逻辑删除过滤、操作者信息、编辑态字段。

这样能避免：

1. 后台查询需求污染前台接口。
2. 权限与字段范围混乱。
3. 一个接口同时服务多端时的条件分支越来越多。

## 数据模型设计

### 1. `sys_user` 表调整

新增字段：

1. `force_password_change boolean not null default false`
2. `password_changed_at datetime null`

字段语义：

1. `force_password_change = true` 表示用户下一次登录后必须先改密。
2. `password_changed_at` 记录最后一次成功修改密码的时间。

规则：

1. 首个系统初始化管理员创建时，`force_password_change = true`。
2. 后台管理员创建新后台账号时，可默认设为 `true`，要求首次登录改密。
3. 调用改密接口成功后，自动将 `force_password_change = false`，并更新 `password_changed_at`。

### 2. `service_order` 表调整

新增字段：

1. `updated_at datetime null`
2. `deleted boolean not null default false`
3. `deleted_at datetime null`
4. `deleted_by bigint null`

规则：

1. 后台删除预约订单时只写逻辑删除标记，不物理删除。
2. 常规列表查询默认过滤 `deleted = true` 的记录。
3. 编辑预约订单时更新 `updated_at`。
4. 删除时写入 `deleted_at` 和当前操作者 `deleted_by`。

### 3. `feedback_submission` 表调整

新增字段：

1. `updated_at datetime null`
2. `deleted boolean not null default false`
3. `deleted_at datetime null`
4. `deleted_by bigint null`

规则与预约订单一致：

1. 删除采用逻辑删除。
2. 常规列表默认过滤逻辑删除数据。
3. 更新时写入 `updated_at`。

### 4. `news_post` 表调整

建议新增字段：

1. `updated_at datetime null`

规则：

1. 动态删除继续采用物理删除，不增加逻辑删除字段。
2. 编辑动态时更新 `updated_at`。

## Redis 设计

验证码与冷却信息不入库，全部存 Redis。

### 1. 验证码会话键

键名：

`auth:captcha:session:{captchaToken}`

建议值：

```json
{
  "scene": "LOGIN",
  "thirdPartyId": "xxxx",
  "clientKey": "sha256(...)",
  "issuedAt": "2026-04-22T20:00:00Z",
  "expiresAt": "2026-04-22T20:05:00Z",
  "used": false
}
```

规则：

1. `captchaToken` 为后端生成的随机字符串。
2. 每个会话只能使用一次。
3. 会话过期后自动删除。
4. 校验成功或进入冷却后，旧会话立即废弃，避免重放。

### 2. 冷却键

键名：

`auth:captcha:cooldown:{scene}:{clientKey}`

值：

冷却截止时间或剩余秒数。

规则：

1. 验证码输入错误后写入 60 秒冷却。
2. 冷却期间同场景请求登录或注册校验时直接拒绝。
3. 冷却结束后必须重新获取新验证码。

### 3. 客户端标识

`clientKey` 使用 `IP + User-Agent 摘要` 生成。

理由：

1. 只按 IP 容易误伤共享网络下多个用户。
2. 只按前端随机标识容易被刷新绕过。
3. `IP + User-Agent` 足以满足当前轻量风控需求。

## 后端接口设计

## 1. 认证相关接口

### `GET /api/auth/captcha?scene=LOGIN|REGISTER`

用途：

1. 获取验证码会话。
2. 若当前客户端仍处于冷却期，则返回剩余秒数并拒绝发新验证码。

响应字段：

1. `captchaToken`
2. `imageUrl`
3. `expireSeconds`
4. `cooldownSeconds`

行为规则：

1. 成功时返回后端图片地址，例如 `/api/auth/captcha/{captchaToken}/image`。
2. 冷却期内返回业务错误码和剩余秒数。

### `GET /api/auth/captcha/{captchaToken}/image`

用途：

1. 根据本地 `captchaToken` 代理验证码图片内容。
2. 前端以后统一加载这个地址，不再使用第三方返回图片链接。

### `POST /api/login`

请求体改为：

```json
{
  "username": "admin",
  "password": "200502",
  "captchaToken": "xxxx",
  "captchaCode": "abcd"
}
```

行为规则：

1. 先检查当前场景是否处于冷却期。
2. 再验证 `captchaToken` 是否存在、未过期、未使用、归属当前客户端。
3. 再调用第三方验证码校验。
4. 若验证码错误，则写入 60 秒冷却并返回剩余时间。
5. 若验证码正确，再校验用户名密码。
6. 若登录成功，返回现有登录信息，并额外返回 `forcePasswordChange`。

### `POST /api/register`

请求体改为：

```json
{
  "username": "user1",
  "password": "123456",
  "region": "xxx",
  "phone": "13800000000",
  "captchaToken": "xxxx",
  "captchaCode": "abcd"
}
```

规则：

1. 只允许普通用户自助注册。
2. 后端不再接受公开注册请求中的 `role = ADMIN`。
3. 注册成功的角色固定为系统允许的普通后台用户角色，如果当前系统仍沿用 `GUARDIAN` 作为普通后台角色，则保持为 `GUARDIAN`。

### `POST /api/account/change-password`

请求体：

```json
{
  "oldPassword": "200502",
  "newPassword": "newPassword123"
}
```

规则：

1. 仅登录用户可调用。
2. 改密成功后更新 `password_hash`、`password_changed_at`、`force_password_change`。
3. 若用户当前处于强制改密态，改密成功后立即解除限制。

调整项：

1. 移除前端直连第三方验证码代码。

页面内容：

2. 列表中增加“编辑”按钮。

4. 编辑后回到列表并刷新。

## 7. 机构管理页面

目标：

1. 所有操作按钮固定为单排。
2. 宽度不足时，优先让表格横向滚动，而不是按钮换行。

实现方向：

1. 操作列使用单行弹性容器，设置 `flex-wrap: nowrap`。
2. 操作按钮统一为 `small` 或 `link` 风格，降低占宽。
3. 操作列宽度明确增大。
4. 若仍过宽，则将次要操作收进“更多”下拉菜单，但主按钮仍单排显示。

推荐优先级：

1. 主按钮：编辑、添加管理员、添加医护、查看人员、启用/禁用
2. 如空间不足，只将“查看人员”收进更多菜单，其他高频动作保持直出

## 权限与安全规则

1. `/api/admin/**` 继续仅允许 `ADMIN` 访问。
2. 自助注册接口不再允许创建 `ADMIN`。
3. 首个 `ADMIN` 默认密码固定为 `200502`，但首次登录必须改密。
4. 验证码错误后 60 秒冷却。
5. 冷却期间不接受同场景验证码校验请求。
6. 验证码会话单次有效，成功或失败进入冷却后即废弃。
7. 管理员改密成功后立即解除强制改密标记。

## 错误处理设计

### 认证相关

1. 验证码会话不存在：返回“验证码已失效，请重新获取”
2. 验证码过期：返回“验证码已过期，请重新获取”
3. 验证码错误：返回“验证码错误，请 60 秒后重试”，同时返回剩余秒数
4. 冷却中：返回“当前操作过于频繁，请 xx 秒后重试”
5. 账号密码错误：沿用现有统一错误信息，不暴露账号存在性以外的额外细节

### 业务相关

1. 编辑已删除的预约或反馈：返回“记录不存在或已删除”
2. 编辑不存在的动态：返回“动态不存在”
3. 非法字段取值：返回结构化校验错误

## 风险与取舍

1. 默认管理员密码固定为 `200502` 本身存在风险，因此必须依赖首次登录强制改密来降低暴露面。
2. 验证码完全依赖第三方服务，若第三方不可用，登录与注册会受影响，因此后端错误提示必须明确，且前端要支持刷新重试。
3. 预约订单与意见反馈新增逻辑删除字段后，旧查询若未统一过滤，容易出现“已删除数据仍可见”的遗漏，因此后台查询必须集中收口。
4. 动态接口若继续与前台接口混用，后续权限容易混乱，因此本次明确后台管理接口单独收口。

## 结论

本设计在不更换验证码供应商的前提下，将后台登录与注册的验证码能力收回后端，并补齐了首个管理员安全初始化、首次登录强制改密、后台 CRUD 缺口和管理界面操作体验问题。它既满足当前提出的安全诉求，也保持了实现范围可控，适合直接进入实施计划阶段。
